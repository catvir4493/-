# 《深夜愿望便利店》AI 开发文档

> 本文档用于给 AI / Codex / Cursor / Claude Code 等开发助手理解项目需求。  
> 项目为全新独立游戏，不关联任何其他旧项目。  
> 推荐引擎：Godot。  
> 推荐开发方式：先做纯 UI 可玩原型，再逐步替换美术、音效和剧情内容。

---

## 0. AI 执行原则

开发 AI 在阅读本文档后，需要遵守以下原则：

1. 优先实现可玩的 MVP，而不是优先追求画面精美。
2. 所有系统都应保持模块化，方便后续扩展商品、顾客、剧情和组合。
3. 第一版不做自由移动、不做战斗、不做大地图。
4. 第一版重点验证核心循环：顾客请求 → 商品选择 → 标签评分 → 顾客反馈 → 获得金钱 → 进货 → 下一夜。
5. 代码结构要清晰，避免把所有逻辑写在一个脚本里。
6. 数据尽量从 JSON 或 Resource 中读取，不要把大量商品和顾客内容硬编码在 UI 脚本里。
7. 所有临时美术可以用占位图、Panel、Label、Button 代替。
8. 系统完成优先级高于美术表现。

# 1. 项目基本信息

## 1.1 游戏名称

中文名：

```text
深夜愿望便利店
```

英文名：

```text
Midnight Wish Mart
```

## 1.2 游戏类型

```text
2D 叙事经营解谜游戏
```

更具体分类：

```text
经营模拟 + 商品组合解谜 + 短篇剧情收集
```

## 1.3 核心概念

玩家经营一家只在凌晨出现的便利店。

进入店里的顾客不会直接说自己要买什么，而是说出一个愿望、烦恼、恐惧或模糊请求。

玩家需要根据顾客的话，从货架上选择 1–3 件商品组合，满足顾客真正的需求。

游戏核心不是传统赚钱经营，而是：

```text
用商品理解人。
```

## 1.4 核心卖点

```text
每个顾客都是一道情绪谜题。
每件商品都有隐藏含义。
不同商品组合会触发不同结果。
重复出现的顾客会推进短篇剧情。
玩家通过经营便利店，逐渐发现这家店为什么只在凌晨出现。
```

## 1.5 目标氛围

```text
深夜
都市
治愈
怪诞
轻悬疑
微黑色幽默
孤独但温暖
```

# 2. 核心玩法设计

## 2.1 一句话玩法

```text
阅读顾客的愿望，从商品中选择合适组合，获得评分、金钱和剧情反馈。
```

## 2.2 单晚流程

每一晚是一个短关卡。

流程如下：

```text
1. 开始营业
2. 顾客进店
3. 顾客说出一句愿望或烦恼
4. 玩家查看货架商品
5. 玩家选择 1–3 件商品
6. 系统计算商品标签是否符合顾客需求
7. 顾客给出反应
8. 玩家获得金钱、评分、剧情线索
9. 下一位顾客进入
10. 本晚所有顾客结束
11. 进入夜晚结算界面
12. 玩家进货、升级、解锁新商品
13. 进入下一夜
```

## 2.3 核心循环

```text
接待顾客 → 理解需求 → 选择商品 → 获得反馈 → 解锁内容 → 继续营业
```

详细拆解：

```text
顾客请求 = 谜题感
商品选择 = 策略感
顾客反馈 = 情绪价值
金钱进货 = 成长感
重复顾客 = 剧情期待
隐藏组合 = 探索动力
```

# 3. MVP 最小可行版本

## 3.1 MVP 目标

第一版只验证一件事：

```text
玩家是否愿意连续接待顾客，并不断尝试商品组合。
```

## 3.2 MVP 必须包含

```text
1. 一个便利店主界面
2. 一个顾客显示区域
3. 一个商品选择界面
4. 20 个商品
5. 30 个顾客请求
6. 一个标签匹配评分系统
7. 一个特殊商品组合系统
8. 一个金钱系统
9. 一个简单进货系统
10. 一个夜晚结算界面
11. 一个本地存档系统
```

## 3.3 MVP 暂时不做

```text
1. 大地图
2. 自由移动
3. 战斗
4. 复杂动画
5. 多结局
6. 完整配音
7. 复杂 AI
8. 联机
9. 成就系统
10. 手机端适配
11. 大量 CG
12. 复杂店铺装修
```

# 4. 推荐技术方案

## 4.1 推荐引擎

```text
Godot
```

推荐原因：

```text
1. 适合 2D 游戏
2. UI 制作方便
3. 节点结构清晰
4. 个人开发成本低
5. 可以快速导出 PC 版本
6. 不需要复杂插件
```

## 4.2 推荐表现形式

MVP 不建议做角色自由移动。

推荐做成固定画面 UI 游戏：

```text
便利店背景 + 顾客立绘 + 商品卡片 + 对话框 + 结算界面
```

这种方式开发量更小，但表现力足够。

## 4.3 画面结构

建议主营业界面结构：

```text
顶部：第几夜 / 当前时间 / 金钱
中间：顾客立绘 / 顾客名称 / 顾客台词
右侧：商品货架
底部：已选择商品 / 确认按钮 / 清空按钮
```

## 4.4 美术风格

```text
2D 像素风 / 手绘风 / UI 卡片风
```

推荐视觉关键词：

```text
暖黄色便利店灯光
深蓝色夜晚背景
雨夜街道
收银台
冰柜微光
孤独城市感
轻微都市传说感
```

## 4.5 音乐音效

MVP 只需要：

```text
1. 深夜便利店循环 BGM
2. 门铃声
3. 收银机声
4. 商品选择声
5. 顾客满意声
6. 顾客不满声
7. 夜晚结算声
```

## 5.3 商品选择系统

### 5.3.1 基本规则

```text
1. 玩家每次面对一位顾客。
2. 玩家可以从货架中选择 1–3 件商品。
3. 选中的商品进入 selected_items。
4. 玩家可以取消选择。
5. 库存为 0 的商品不可选择。
6. 提交后扣除对应商品库存。
7. 提交后进入评分系统。
```

### 5.3.2 商品选择限制

```text
最少选择：1 件商品
最多选择：3 件商品
```

### 5.3.3 UI 反馈

```text
已选择商品高亮显示。
库存不足商品置灰。
超过 3 件商品时提示：最多只能选择 3 件商品。
未选择商品时点击确认，提示：请至少选择 1 件商品。
```

## 5.4 标签评分系统

### 5.4.1 基本逻辑

玩家选择商品后，系统合并所有商品标签，然后和当前顾客的需求标签进行匹配。

评分由三部分组成：

```text
1. 命中 required_tags
2. 避开 avoid_tags
3. 是否触发特殊组合
```

### 5.4.2 评分规则

```text
每命中一个 required_tag：+20 分
每出现一个 avoid_tag：-20 分
每件商品基础适配：+5 分
触发特殊组合：+20 分
商品数量过多但无有效标签：-10 分
```

### 5.4.3 评分等级

```text
90–100：perfect / 完美
70–89：good / 满意
40–69：normal / 一般
0–39：fail / 失败
```

### 5.4.4 评分伪代码

```pseudo
score = 0
selected_tags = []

for item in selected_items:
    selected_tags += item.tags
    score += 5

for tag in selected_tags:
    if tag in customer.required_tags:
        score += 20

    if tag in customer.avoid_tags:
        score -= 20

combo_result = ComboSystem.check(selected_items)
if combo_result != null:
    score += combo_result.score_bonus

score = clamp(score, 0, 100)

grade = get_grade(score)
return score, grade, matched_tags, bad_tags, combo_result
```

### 5.4.5 顾客反馈示例

```text
perfect：
“谢谢。我好像知道该怎么做了。”

good：
“这应该有用。至少今晚有用。”

normal：
“也许吧……我会试试。”

fail：
“你根本没听懂我在说什么。”
```

## 5.6 金钱系统

### 5.6.1 金钱来源

玩家通过接待顾客获得收入。

收入由以下因素决定：

```text
1. 商品售价
2. 顾客评分
3. 是否触发特殊组合
4. 是否完成剧情顾客目标
```

### 5.6.2 收入公式

MVP 使用简单公式：

```pseudo
final_income = selected_items_sell_price_total + score_bonus
```

score_bonus：

```text
perfect：+10
good：+5
normal：+0
fail：-2
```

### 5.6.3 失败惩罚

失败时不要让玩家直接卡死。

建议：

```text
失败只扣少量钱或不给奖励。
不要让玩家因为一次失败无法继续游戏。
```

# 6. 剧情设计

## 6.1 世界观

这是一家只在凌晨 0:00 到 4:00 出现的便利店。

它不在地图上，也没有固定地址。

只有真正需要某样东西的人，才会在深夜走进这里。

店里的商品看起来普通，但每一件都带有某种情绪或意义。

玩家是这家店的新任夜班店员。

玩家不知道上一任店员去了哪里。

## 6.2 剧情推进方式

剧情不通过长篇动画推进，而是通过以下方式展开：

```text
1. 顾客重复出现
2. 顾客档案更新
3. 商品描述变化
4. 夜晚广播出现奇怪内容
5. 收银台留下纸条
6. 某些商品自动补货
7. 店门外街景逐渐变化
```

## 6.3 主线悬念

```text
1. 便利店为什么只在凌晨出现？
2. 顾客为什么能找到这里？
3. 上一任店员去了哪里？
4. 店里的商品到底从哪里来？
5. 玩家自己为什么会成为夜班店员？
```

## 6.4 MVP 剧情目标

第一版只需要做到：

```text
1. 玩家知道这是奇怪便利店
2. 玩家接待一批深夜顾客
3. 出现 2–3 个重复顾客
4. 第 5 夜出现一个特殊顾客
5. 结尾留下悬念
```

## 6.5 第 5 夜特殊事件

第 5 夜最后一位顾客是“上一任店员”。

示例台词：

```text
上一任店员：“我想买一样东西。”
玩家：“您需要什么？”
上一任店员：“我想买回我离开这里的理由。”
```

玩家需要选择商品组合。

不同组合会触发不同结尾提示。

请帮我实现商品选择系统。

# 16. 开发重点提醒

这个游戏最重要的不是系统复杂，而是：

```text
顾客台词要有吸引力。
商品描述要有想象空间。
组合结果要让玩家觉得“原来还可以这样”。
每晚流程要短。
每次结算都要给玩家一点新东西。
```

不要一开始追求大而全。

先做出一个能玩 10 分钟、让人想继续试商品组合的小版本。

只要这个核心循环成立，后面加商品、顾客、剧情，游戏就能自然变大。
